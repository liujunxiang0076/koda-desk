# Koda Desk 需求文档

## 1. 项目概述

Koda Desk 是一个轻量桌面宠物应用。应用使用 Tauri 提供桌面窗口、系统托盘、配置和系统交互能力，前端使用 Canvas 渲染 spritesheet 宠物动画。

第一版目标是做出一个稳定、低资源占用、可长期常驻桌面的宠物 MVP：窗口透明无边框、宠物置顶显示、播放 idle 动画、可拖拽移动，并通过右键菜单和系统托盘进行基础控制。

后续版本在 MVP 稳定后，再扩展宠物切换、状态识别、状态动画、点击反馈、缩放、位置记忆和开机自启等能力。

## 2. 产品目标

### 2.1 MVP 目标

- 桌面上只显示宠物本体，不显示普通应用 UI。
- 宠物窗口透明、无边框、默认置顶。
- 宠物从 spritesheet 资源中读取并播放 idle 动画。
- 用户可以拖拽宠物移动位置。
- 用户可以通过右键菜单隐藏、显示、退出应用。
- 用户可以通过系统托盘入口控制宠物。
- 应用常驻时资源占用低，不明显影响日常使用。

### 2.2 非目标

MVP 暂不实现以下功能：

- 多宠物切换。
- 自动识别用户工作状态。
- 多状态动画自动切换。
- 点击宠物触发反应。
- 宠物缩放。
- 位置记忆。
- 开机自启。
- 完整设置页。

这些功能进入第二阶段或后续版本。

## 3. 用户体验

### 3.1 默认体验

- 应用启动后，在桌面显示一个透明背景宠物窗口。
- 窗口不展示标题栏、边框、按钮或普通 UI。
- 宠物默认播放 idle 动画。
- 宠物窗口默认保持置顶。
- 鼠标可以拖动宠物窗口改变位置。
- 用户可以通过右键菜单或系统托盘执行基础操作。

### 3.2 右键菜单

MVP 右键菜单包含：

- 隐藏宠物。
- 显示宠物。
- 退出应用。

说明：

- 当宠物已显示时，显示宠物项可以禁用或保持可点击但无副作用。
- 当宠物已隐藏时，用户主要通过系统托盘重新显示宠物。

### 3.3 系统托盘

MVP 托盘至少包含：

- 显示/隐藏宠物。
- 退出。

第二阶段托盘菜单扩展为：

- 显示/隐藏宠物。
- 切换宠物。
- 缩放：小 / 中 / 大。
- 状态：自动 / 摸鱼 / 工作。
- 设置。
- 退出。

### 3.4 设置入口

MVP 可以不做完整设置页，但需要预留设置入口设计。第二阶段开始可以通过以下方式打开设置：

- 托盘菜单点击“设置”。
- 独立设置窗口。
- 设置窗口不影响宠物主窗口透明、置顶、无边框的特性。

## 4. 功能需求

### 4.1 透明无边框窗口

要求：

- 宠物主窗口透明。
- 无标题栏、无系统边框。
- 窗口背景不显示默认颜色。
- 主窗口尺寸尽量贴合宠物渲染区域。

验收：

- 启动后只看到宠物，不看到矩形窗口背景。
- 透明区域不出现黑底、白底或系统默认背景。
- 窗口没有标题栏和边框。

### 4.2 宠物置顶显示

要求：

- 宠物主窗口默认 always on top。
- 置顶行为不应抢占键盘焦点。
- 置顶行为不应明显干扰其他应用的正常使用。

验收：

- 切换到其他应用后，宠物仍显示在桌面上方。
- 其他应用仍可正常输入和操作。

### 4.3 Canvas 播放 idle 动画

要求：

- 前端使用 Canvas 渲染 spritesheet。
- MVP 默认读取 idle 动画配置并循环播放。
- 动画帧率目标为 8-12 FPS。
- 支持透明背景帧。

验收：

- 宠物启动后能连续播放 idle 动画。
- 动画循环自然，没有明显错帧、空白帧或透明背景异常。
- 空闲状态下 CPU 占用保持较低。

### 4.4 拖拽移动

要求：

- 用户按住宠物区域可拖拽移动窗口。
- 拖拽过程应流畅。
- MVP 可以不记忆位置，重启后使用默认位置。
- 第二阶段增加位置记忆。

验收：

- 鼠标按住宠物后，窗口跟随鼠标移动。
- 松开鼠标后，宠物停留在当前位置。
- 拖拽时不触发异常菜单或卡死。

### 4.5 右键菜单控制

要求：

- 用户右键宠物区域可打开菜单。
- 菜单提供隐藏、显示、退出。
- 退出需要关闭宠物窗口并结束托盘进程。

验收：

- 点击隐藏后宠物窗口不可见。
- 点击显示后宠物窗口重新出现。
- 点击退出后应用完全退出，托盘图标消失。

### 4.6 系统托盘入口

要求：

- 应用启动后创建系统托盘图标。
- 托盘菜单提供显示/隐藏和退出。
- 宠物隐藏后，用户仍能通过托盘恢复显示。

验收：

- 系统托盘能看到应用入口。
- 托盘菜单能正常打开。
- 隐藏宠物后，托盘仍存在。
- 退出后托盘图标移除。

### 4.7 读取宠物资源

要求：

- 资源包含 `spritesheet.webp` 和 `pet.json`。
- MVP 至少支持读取一个默认宠物。
- 资源格式继续兼容 Codex 宠物资源格式。
- 如果资源加载失败，应有基础错误处理，不能导致应用无响应。

验收：

- 默认宠物资源能成功加载。
- 删除或损坏资源时，应用不会卡死。
- 控制台或日志能看到明确的资源加载错误。

## 5. 宠物资源格式

### 5.1 spritesheet 规格

- 文件名：`spritesheet.webp`。
- 尺寸：1536x1872。
- 网格：8 列 x 9 行。
- 单格：192x208。
- 背景：透明。

### 5.2 pet.json 基础结构

`pet.json` 需要描述宠物基础信息、spritesheet 布局和动画状态。建议 MVP 使用如下结构：

```json
{
  "name": "default",
  "version": "1.0.0",
  "spritesheet": "spritesheet.webp",
  "frame": {
    "width": 192,
    "height": 208,
    "columns": 8,
    "rows": 9
  },
  "animations": {
    "idle": {
      "frames": [0, 1, 2, 3],
      "fps": 8,
      "loop": true
    }
  }
}
```

### 5.3 帧索引规则

- 帧索引从 0 开始。
- 按从左到右、从上到下计算。
- 第 0 行第 0 列为 0。
- 第 0 行第 7 列为 7。
- 第 1 行第 0 列为 8。

### 5.4 第二阶段状态动画

后续可在 `animations` 中补充：

- `idle`：摸鱼 / 发呆。
- `working`：疯狂敲代码。
- `waiting`：等待确认。
- `failed`：崩溃趴键盘。
- `review`：认真检查。
- `dragging`：被拖动时左右跑。

示例：

```json
{
  "animations": {
    "idle": {
      "frames": [0, 1, 2, 3],
      "fps": 8,
      "loop": true
    },
    "working": {
      "frames": [8, 9, 10, 11],
      "fps": 12,
      "loop": true
    }
  }
}
```

## 6. 技术方案

### 6.1 技术栈

- 桌面壳：Tauri。
- 前端渲染：Canvas。
- 动画资源：spritesheet + `pet.json`。
- 系统能力：Rust / Tauri API。
- 配置与状态：Tauri 后端管理，前端读取必要状态。

### 6.2 前端职责

- 加载 `pet.json`。
- 加载 `spritesheet.webp`。
- 使用 Canvas 绘制当前动画帧。
- 控制动画循环、FPS 和状态切换。
- 处理宠物区域的鼠标事件。
- 将拖拽、右键菜单等事件转交给 Tauri。

### 6.3 Rust / Tauri 职责

- 创建透明无边框窗口。
- 设置窗口置顶。
- 控制窗口显示、隐藏和退出。
- 创建和维护系统托盘。
- 提供右键菜单能力。
- 管理配置文件。
- 后续对接系统状态识别、开机自启等能力。

### 6.4 建议目录结构

```text
koda-desk/
  src/
    main.ts
    renderer/
      petCanvas.ts
      animation.ts
      petManifest.ts
    styles/
      app.css
  src-tauri/
    src/
      main.rs
      tray.rs
      window.rs
      config.rs
    tauri.conf.json
  public/
    pets/
      default/
        spritesheet.webp
        pet.json
  docs/
    requirements.md
```

实际开发时可根据 Tauri 模板调整，但应保持宠物渲染、动画配置、窗口控制、托盘控制职责清晰。

## 7. 状态设计

### 7.1 MVP 状态

MVP 只需要支持：

- `idle`：默认循环播放。

### 7.2 第二阶段状态

第二阶段支持：

- `idle`：摸鱼 / 发呆。
- `working`：疯狂敲代码。
- `waiting`：等待确认。
- `failed`：崩溃趴键盘。
- `review`：认真检查。
- `dragging`：被拖动时左右跑。

### 7.3 状态切换来源

后续状态切换可以来自：

- 用户托盘菜单手动选择。
- 点击宠物触发临时状态。
- 拖拽宠物时进入 `dragging`。
- 根据用户工作状态自动判断。
- 外部应用或插件传入状态。

### 7.4 自动状态识别初步设想

第二阶段可先做轻量识别：

- 前台窗口标题或应用名。
- 最近键盘 / 鼠标活动频率。
- IDE、终端、浏览器等应用活跃度。
- 用户手动设置“自动 / 摸鱼 / 工作”模式。

自动识别必须可关闭，并允许用户手动覆盖。

## 8. 配置设计

MVP 可先使用默认配置。第二阶段建议支持：

```json
{
  "pet": {
    "current": "default",
    "scale": "medium"
  },
  "window": {
    "x": null,
    "y": null,
    "alwaysOnTop": true
  },
  "behavior": {
    "mode": "auto",
    "state": "idle"
  },
  "startup": {
    "launchOnBoot": false
  }
}
```

配置要求：

- 配置文件不存在时使用默认值。
- 配置损坏时回退默认值，并记录错误。
- 位置记忆、缩放、宠物切换等功能必须通过配置持久化。

## 9. 性能目标

- 常驻内存尽量低。
- 空闲时 CPU 占用低。
- 动画帧率控制在 8-12 FPS。
- 不使用高频无意义渲染。
- 窗口尺寸小，不占桌面空间。
- 不影响鼠标正常使用。
- 资源加载失败或动画暂停时不应持续占用 CPU。

建议：

- 使用固定 FPS 的渲染循环。
- 页面不可见或宠物隐藏时暂停 Canvas 动画。
- 避免每帧解析配置。
- spritesheet 加载后复用图片对象。

## 10. 兼容性与平台

MVP 优先支持：

- Windows。

后续可扩展：

- macOS。
- Linux。

需要重点验证：

- 透明窗口在 Windows 下是否正确。
- 无边框置顶窗口是否影响输入焦点。
- 托盘菜单在 Windows 下是否稳定。
- WebView 透明背景是否正确启用。

## 11. 日志与错误处理

MVP 至少需要覆盖：

- 宠物资源加载失败。
- `pet.json` 解析失败。
- spritesheet 图片加载失败。
- 托盘初始化失败。
- 窗口显示 / 隐藏失败。

错误处理原则：

- 不让应用无响应。
- 尽量保留托盘入口，方便用户退出。
- 开发阶段可输出到控制台。
- 后续版本增加日志文件或诊断页面。

## 14. 风险与待确认事项

### 14.1 技术风险

- Tauri 透明窗口在不同平台行为不完全一致。
- WebView 透明背景可能需要额外配置。
- 无边框窗口拖拽可能需要区分左键拖拽和右键菜单。
- 置顶窗口可能影响用户日常操作，需要避免抢焦点。
- 托盘 API 在 Tauri 版本之间可能有差异。
- 透明窗口点击穿透是否需要支持，MVP 暂不要求。

### 14.2 产品待确认

- MVP 默认宠物资源由谁提供。
- 宠物默认出现位置。
- 宠物是否需要支持点击穿透。
- 宠物窗口是否允许显示在任务栏。
- 右键菜单是使用 Tauri 原生菜单，还是前端自绘菜单。
- 隐藏后再次显示的位置是否保持在本次运行时的最后位置。

## 15. 后续维护原则

- 先保证 MVP 稳定，再扩展状态能力。
- 宠物资源格式保持兼容 Codex spritesheet 规范。
- 状态识别必须可关闭，避免用户感觉被打扰。
- 桌面宠物应低打扰、低资源占用。
- 每次新增行为都要检查是否影响透明、置顶、托盘和退出这几个核心路径。
