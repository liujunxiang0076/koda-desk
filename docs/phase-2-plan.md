# 第二阶段后续任务计划

## 1. 文档目的

本文档记录 Koda Desk 第二阶段的后续动作任务，用于承接 `docs/requirements.md` 中 `13.2 第二阶段拆分` 的开发计划。

当前前三项状态：

| 序号 | 任务 | 状态 | 说明 |
| --- | --- | --- | --- |
| 1 | 增加宠物资源目录扫描 | 已完成一版 | 前端设置页、初始宠物校验和 Rust 托盘宠物菜单都复用 `public/pets/pets.json`，列表读取失败时仍保留 `koda` 兜底。 |
| 2 | 增加宠物切换 | 已完成一版 | 设置页和托盘已有宠物切换入口，托盘切换会重新显示宠物窗口并同步前端状态。 |
| 3 | 增加缩放配置 | 已完成一版 | 支持小 / 中 / 大缩放和配置保存，窗口尺寸跟随宠物实际帧尺寸；仍需真实 Tauri 桌面复测。 |

## 2. 后续任务清单

### 2.1 补完整宠物资源注册

状态：已完成一版；后续新增宠物仍需同步补资源目录和 manifest。

- 以 `public/pets/pets.json` 作为唯一宠物列表来源。
- 去掉前端 `supportedPets` 中的 `koda` / `lumen` 硬编码。
- 让初始宠物校验、设置页选项、宠物切换都复用同一份注册表。
- 宠物列表加载失败时保留可控兜底，避免应用不可退出。

验收：

- 新增宠物只需要更新 `public/pets/pets.json` 和对应资源目录。
- 前端不会因为硬编码宠物列表漏掉新宠物。
- 非法宠物名称不会导致应用崩溃。

### 2.2 同步托盘宠物菜单

状态：已完成一版；Rust 托盘宠物菜单已从同一份宠物注册表生成。

- 梳理 Rust 托盘菜单与前端宠物列表的数据来源。
- 将托盘宠物菜单从硬编码 `Koda` / `Lumen` 迁移为可配置来源。
- 确认托盘选择宠物后能显示宠物窗口并同步前端状态。

验收：

- 托盘宠物菜单与设置页宠物列表一致。
- 从托盘切换宠物后，设置页选中项同步更新。
- 宠物隐藏状态下，从托盘切换宠物会重新显示宠物。

### 2.3 验证第二阶段前三项

状态：自动检查已通过；真实 Tauri 桌面手测仍需执行。

- 启动应用并确认默认宠物加载。
- 在设置页切换宠物。
- 在托盘切换宠物。
- 切换小 / 中 / 大缩放。
- 确认窗口尺寸跟随缩放变化。
- 确认切换宠物后窗口尺寸按当前宠物帧尺寸恢复。
- 确认关闭设置页后恢复到正确宠物窗口尺寸。
- 确认配置保存后重启仍生效。

验收：

- 宠物切换、缩放、设置页关闭、重启恢复没有明显状态错乱。
- `npm run build` 通过。
- `cargo check` 通过。
- Tauri 桌面手测无 P0 / P1 问题。

### 2.4 进入第 4 项：位置记忆

状态：已完成一版；拖拽结束会保存位置，启动恢复时会按可用屏幕边界钳制，仍需真实桌面拖拽 / 重启复测。

- 梳理当前已有的 `save_window_position`、`window.x/y`、启动恢复逻辑。
- 补齐拖拽结束后的保存时机。
- 确认重启后窗口恢复到上次位置。
- 增加屏幕边界保护，避免窗口恢复到屏幕外。

验收：

- 拖拽宠物后会保存位置。
- 关闭并重启应用后，宠物出现在上次位置。
- 显示器分辨率或屏幕布局变化时，窗口不会恢复到不可见区域。

### 2.5 进入第 5 项：状态机

状态：已完成一版；前端新增统一状态控制器，临时状态、手动状态和自动状态走单一出口。

- 定义状态来源优先级：拖拽临时状态、点击临时状态、手动状态、自动状态。
- 避免多个事件处理器直接调用 `pet.setState(...)` 互相覆盖。
- 抽出统一状态管理模块。
- 明确状态回退策略。

验收：

- 拖拽、点击、手动状态、自动状态之间切换稳定。
- 临时状态结束后能回到正确的基础状态。
- 状态切换逻辑有单一入口，便于后续扩展。

### 2.6 进入第 6 项：多状态动画

状态：已完成一版；`koda`、`lumen`、`default` manifest 均补齐多状态动画配置，动画帧和 fps 有校验 / 安全限制。

- 扩展 `pet.json` 动画校验。
- 支持缺失状态动画时回退 `idle`。
- 对非法帧索引输出明确错误。
- 对异常 `fps` 做安全限制或明确失败处理。

验收：

- `working`、`waiting`、`failed`、`review`、`dragging` 等状态可按配置播放。
- 状态动画缺失时不崩溃。
- 越界帧和异常 `fps` 不会造成高 CPU 或空白循环。

### 2.7 进入第 7 项：点击反应

状态：已完成一版；点击进入短暂 `review`，拖拽进入临时 `dragging`，并通过拖拽位移避免拖拽后误触发点击反应。

- 将当前点击临时进入 `review` 的逻辑整理为可配置行为。
- 定义点击反应状态和持续时间。
- 避免点击与拖拽事件互相误触发。

验收：

- 单击宠物能触发短暂反应。
- 拖拽宠物不会误触发点击反应。
- 点击反应结束后回到正确状态。

### 2.8 进入第 8 项：工作状态识别

状态：已完成轻量版；自动模式按窗口可见性和本地时间推断 `waiting` / `working` / `idle`，手动模式优先级更高。

- 先实现轻量自动判断。
- 保证自动识别可关闭。
- 保证用户手动状态优先级高于自动识别。
- 不做侵入式监听，避免高打扰。

验收：

- 自动模式能在基础条件下切换工作 / 摸鱼状态。
- 手动模式不会被自动识别覆盖。
- 自动识别失败时应用仍保持可用。

### 2.9 完善设置页

状态：已完成一版；宠物、缩放、模式、状态和开机自启会保存配置，设置页打开时暂停动画、关闭后恢复宠物窗口尺寸。

- 将宠物、缩放、模式、状态、开机自启配置都稳定保存。
- 设置页打开和关闭不影响宠物主窗口透明、置顶、无边框路径。
- 托盘打开设置时会主动显示主窗口，避免隐藏状态下设置页不可见。
- 设置页关闭后由前端按当前宠物和缩放统一恢复窗口尺寸，避免与 Rust 固定尺寸竞争。

验收：

- 设置项保存后重启仍生效。
- 设置页关闭后宠物窗口恢复正确尺寸。
- 设置页不会破坏宠物主窗口的核心显示行为。
- 开机自启勾选状态能同步系统实际注册状态。

### 2.10 阶段验收与记录

状态：自动构建、Rust 检查和开机自启插件编译已记录；真实 Tauri 桌面交互、托盘、系统登录自启和资源异常场景仍需专项复测。

- 更新 `docs/mvp-validation.md`，或新增第二阶段验证记录。
- 执行 `npm run build`。
- 执行 `cargo check`。
- 做 Tauri 桌面手测。
- 汇总 P0 / P1 / P2 / P3 问题。
- 根据结果决定是否进入更完整设置页和打包发布前验证。

验收：

- 第二阶段核心路径有明确测试记录。
- 已知 P0 / P1 问题清零。
- 未完成项有后续归属和优先级。

## 3. 建议执行顺序

1. 已补完整宠物资源注册。
2. 已同步托盘宠物菜单。
3. 已完成宠物切换和缩放配置的自动检查，仍需真实 Tauri 桌面复测。
4. 已实现位置记忆和屏幕边界保护，仍需拖拽 / 重启复测。
5. 已抽出状态机。
6. 已完善多状态动画配置和校验。
7. 已整理点击反应和拖拽误触发保护。
8. 已增加轻量工作状态识别。
9. 已收口现有设置页保存、开关行为和开机自启入口。
10. 已新增阶段验收记录，真实桌面手测和系统登录自启结果待补充。

## 4. 风险提示

- 托盘菜单由 Rust 创建，宠物清单位于前端 `public/` 资源目录，两侧数据来源需要谨慎统一。
- 位置记忆需要处理屏幕边界，否则可能导致宠物窗口恢复到屏幕外。
- 状态机上线前，应减少散落的直接 `pet.setState(...)` 调用，避免状态覆盖问题继续扩大。
- 自动工作状态识别必须可关闭，并且不能影响用户正常操作。
- 开机自启需要在安装后的真实系统登录 / 重启路径中复测，单靠编译检查不能证明系统注册项一定生效。
