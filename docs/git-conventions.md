# Git 提交规范

## 提交粒度

- 一次提交只表达一个清晰目的，例如功能、修复、文档、构建配置或重构。
- 功能代码和纯文档规范可以分开提交，便于回滚和审查。
- 提交前确认 `git status --short --branch`，不要混入临时文件、构建产物、依赖目录或无关改动。

## 提交信息格式

使用 Conventional Commits：

```text
<type>: <中文动宾短句>
```

常用类型：

| 类型 | 用途 |
| --- | --- |
| `feat` | 新功能或用户可感知能力 |
| `fix` | 缺陷修复或行为纠偏 |
| `docs` | 文档、规范、说明 |
| `chore` | 仓库维护、依赖、生成文件、非产品行为改动 |
| `refactor` | 不改变行为的代码整理 |
| `test` | 测试补充或测试设施 |
| `build` | 构建系统、打包配置、CI |

示例：

```text
feat: 统一宠物注册表来源
docs: 补充 Git 提交规范
fix: 修正设置页状态联动
```

## 提交前检查

至少执行：

```powershell
git diff --cached --check
npm run build
cargo check
```

如果某项检查无法执行，需要在交付说明中写明原因和风险。

## 推送规范

- 默认推送当前分支到同名远端分支。
- 推送前确认本地分支与远端关系：`git status --short --branch`。
- 不使用强制推送，除非用户明确要求并确认影响范围。
